<Section eyebrow="Context" title="Why this exists">

Life In Character is a personal engineering build — not a product. It's a Habitica-inspired RPG habit tracker I built to push on three things: a feature-based Redux architecture large enough to feel real, a sprite-layered avatar system with hundreds of assets, and the gamification mechanics (XP, gold, health, daily reset) that make Habitica work.

The sprite assets are from the open-source [Habitica](https://habitica.com) project, licensed **CC-BY-NC-SA 3.0**. That license is the reason this app can't be commercialized — and it's also the reason the case study exists in the first place. It freed me to build the thing honestly as a study, without product pressure.

</Section>

<Section eyebrow="Key decisions" title="Interesting calls">

<Decisions items={[
  {
    title: "Feature-based Redux slices, not a single store",
    body: "Six separate slices — habits, dailies, todos, player, character, inventory — each with their own selectors and reducers. Easier to reason about, easier to test in isolation, and surfaced a real insight: player state (XP, gold, health) needs to react to events from three other slices, so thunks became the clear boundary.",
  },
  {
    title: "Redux Persist + AsyncStorage, no backend",
    body: "Everything lives on-device. No account system, no cloud sync. The build rule was 'if it doesn't need a server, it doesn't get one' — which kept the project scope honest and let me focus on the mechanics rather than the plumbing.",
  },
  {
    title: "Three task archetypes from Habitica's model",
    body: "Habits (flexible positive/negative), Dailies (scheduled with streaks), Todos (one-shot). Tried collapsing them during prototyping — regretted it. The three archetypes exist because they model genuinely different behaviors; merging them made the UI worse, not simpler.",
  },
  {
    title: "Sprite map with 460+ keys, layered avatar renderer",
    body: "The avatar is a stack of sprite layers — skin, hair, shirt, armor, helmet, weapon, pet — each driven by equipment state in the inventory slice. CharacterAvatar.tsx composes the layers in Z-order so equipping an item updates the render automatically.",
  },
  {
    title: "Daily reset with missed-task penalties",
    body: "Dailies lose health when missed, which is the single most motivating mechanic in Habitica. Implementing it meant a scheduled check on app open: diff last-reset timestamp against today, apply penalties, reset streaks. Small feature, huge behavior impact.",
  },
]} />

</Section>

<Section eyebrow="What I learned" title="Honest outcome">

The biggest lesson was about scope. Habitica has been shipping for over a decade; my version is a system study, not a rival. Keeping that straight meant resisting feature creep — I don't need parties, challenges, or sync, because the point was to understand how the mechanics hold together, not to compete.

The sprite-layering system was the unexpected win. Building it forced a clean separation between "what's equipped" (inventory slice) and "what's rendered" (presentation layer), and that separation turned out to be the cleanest architecture decision in the project.
